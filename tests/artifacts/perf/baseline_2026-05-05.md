## Analysis

### Primary Hotspot: ufunc_add_broadcast
The broadcast add operation (1024x1024 by 1024) takes 16.85ms to process 1M elements,
yielding only 62.3M elem/s. In contrast, reduce_sum processes the same 1M elements
in 766µs (1.37B elem/s) - a 22x difference.

**Hypothesis:** Broadcasting overhead dominates. The broadcast loop iterates per-element
instead of using SIMD-friendly contiguous access patterns.

**Recommended investigation:**
1. Profile with perf/samply to identify exact hotspot within elementwise_binary
2. Check if broadcast shape calculation is repeated per-element
3. Consider specialized fast path for common broadcast patterns (row vector, col vector)
