# Fixture Provenance

> Mandatory per the testing-conformance-harnesses skill. Every fixture records
> *how* it was generated: tool, NumPy version, command, and last regeneration
> date. When a regenerated fixture disagrees with the committed copy, this
> table is the first place to look.

## Environment baseline

- **Reference NumPy:** 2.4.3 (installed at `/home/ubuntu/.local/lib/python3.13/site-packages/numpy`)
- **Reference Python:** 3.13.7
- **Oracle capture binary:** `cargo run --bin capture_numpy_oracle -p fnp-conformance`
- **Oracle Python resolver:** honors `$FNP_ORACLE_PYTHON`; otherwise probes `python3` on `$PATH`
- **Pure-Python fallback:** used when no NumPy interpreter is available; gate with `FNP_REQUIRE_REAL_NUMPY_ORACLE=1` to forbid fallback in CI
- **Repository root at last regen:** `git rev-parse HEAD` at each fixture's last-regen date below
- **Fixture ID policy:** IDs must be snake_case lowercase; enforced by `validate_fixture_id`

## Generator key

- `capture_numpy_oracle` — runs `capture_numpy_oracle` on the matching `*_input_cases.json` or directly against live NumPy; writes a normalized `cases[]` record per input
- `manual` — hand-authored cases, typically tabulated adversarial / boundary / IEEE-754 scenarios that do not come from a live oracle
- `programmatic` — generated by a Rust test helper or a one-off script in `src/` that enumerates a cartesian product (e.g. dtype × dtype) and records expected values
- `oracle_capture_script` — a specific generator function inside `src/<domain>.rs`; see referenced lines

## Regeneration workflow

For `capture_numpy_oracle` fixtures:

```bash
export FNP_ORACLE_PYTHON=/home/ubuntu/.local/bin/python3  # NumPy-backed
export FNP_REQUIRE_REAL_NUMPY_ORACLE=1                    # no silent fallback
cargo run -p fnp-conformance --bin capture_numpy_oracle
# captures ufunc_input_cases.json → oracle_outputs/ufunc_oracle_output.json
```

For `manual` fixtures: edit the JSON directly, rerun the affected differential
test, and update `Last regen` in this table.

For `programmatic` fixtures (dtype_promotion_cases.json): rerun the helper
referenced in its `generator` field. Record the exact command in this file.

## Upgrade protocol

When upgrading the reference NumPy version (e.g. 2.4.3 → 2.5.x):

1. Update the `Reference NumPy` line at the top of this file.
2. Run `cargo run -p fnp-conformance --bin capture_numpy_oracle` for every
   `capture_numpy_oracle` fixture listed above.
3. `git diff fixtures/` — review every changed row. File a bead for each
   divergence that is *not* explained by the NumPy changelog.
4. Update each affected fixture's `Last regen` row in this file.
5. If a divergence is an intentional NumPy-side bugfix that we should match,
   the diff is accepted — bump the `NumPy` column. If it is an intentional
   fnp divergence, document it in `../DISCREPANCIES.md` with a new `DISC-NNN`
   ID and set the test to XFAIL.
